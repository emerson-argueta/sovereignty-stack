Chapter 11

The System Model — Primitives & Enforcement

Sovereignty as an architectural property requires enforcement points, not just policy declarations. This chapter defines the hard primitives, system layers, and request lifecycle that transform the framework from a meta-model into a reference architecture.

The preceding chapters describe what sovereignty means at each layer. This chapter describes how it is enforced. The distinction is the difference between a constitution that declares rights and a court system that protects them. Declarations without enforcement are philosophy. Enforcement without declaration is arbitrary power. The sovereign system requires both — and this chapter specifies the enforcement half.

From Meta-Model to Reference Architecture

Every previous chapter described sovereignty in terms of layers: physical, institutional, cognitive, knowledge, geopolitical. These are the right categories. But categories are not systems. A system requires:

Primitives — the irreducible data types everything else is built from. If you cannot define sovereignty in five or fewer primitives, you have not understood it.

Enforcement points — the specific locations in the system where control is actually exercised. Sovereignty does not live in documents. It lives in the places where requests are allowed or denied.

A request lifecycle — the complete sequence from input to output, with every step mapped to an enforcement point and every enforcement point mapped to a framework layer. Without this, you cannot answer: what happens when someone tries to do something they are not authorized to do?

The Diagnostic Test

If you cannot point to a specific place in the system where a particular policy is enforced — not where it is declared, not where it is documented, but where it is executed — that policy does not exist in the system. It exists only in the documentation of the system. Governance is a runtime property, not a design-time property.

The Five Hard Primitives

Every operation in a sovereign system reduces to combinations of five primitives. These are not implementation choices — they are logical necessities. Any system that cannot express itself in these terms is either incomplete or not actually a system.

Primitive 01 — Identity
Identity { domain_id: bytes // pairwise: KDF(master_secret, domain_context) pubkey: bytes // hardware-bound signing key, domain-scoped claims: Claim[] // verifiable credentials: citizenship, role, clearance lifecycle: { issued, expires, revoked_at, successor } // cross-domain linkage requires explicit DataFlowPolicy authorization }

Identity is not a username. It is a cryptographic binding between a real actor and a verifiable set of claims, scoped to the domain of interaction. The domain_id is derived per-domain: the same actor presents a different, unlinkable identifier in each context. Cross-domain correlation — linking the same person's activity across domains — requires explicit DataFlowPolicy authorization and is recorded in the audit log. A global reusable identifier is a surveillance primitive; the sovereign system does not issue them. Identity without a hardware-bound key is spoofable; spoofable identity is not identity.

Primitive 02 — Resource
Resource { id: string // globally unique type: enum // dataset | model | pipeline | infrastructure | key owner: Identity // who controls access policy classification: enum // public | restricted | sovereign | classified lineage: Resource[] // what this resource was derived from }

A Resource is anything the system must protect. Energy infrastructure, financial rails, training datasets, model weights, cryptographic keys — all are Resources. Lineage is critical: a model trained on a classified dataset inherits its classification. A pipeline that processes restricted data cannot produce public outputs without explicit declassification policy.

Primitive 03 — Policy
Policy { subject: Identity | Role | Group action: enum // read | write | execute | delegate | admin resource: Resource | Classification | Type condition: Context → boolean // time, location, quorum, threshold effect: allow | deny | audit | require_mpc }

Policy is the formalization of governance. It is not a document — it is a function. Given a subject, an action, a resource, and the current context, the policy evaluator returns a decision. Policies are versioned, audited, and themselves governed by higher-order policies. The governance of governance is where sovereignty is most fragile and most important.

Primitive 04 — Execution Context
ExecutionContext { request_id: string // globally unique, unforgeable initiator: Identity // who originated the request provenance: AuditEntry[] // full chain of prior actions that led here environment: { node, hardware, attestation, timestamp } constraints: Policy[] // policies active at time of execution }

Every computation in the system runs inside an Execution Context. The context carries provenance: the complete chain of actions that produced the current state. A computation whose context cannot be verified was not authorized. A computation whose environment cannot be attested may have been tampered with. Execution without context is outside the system boundary — which means it is outside sovereignty's reach.

Primitive 05 — Audit Log
AuditEntry { entry_id: string // hash of prior entry + content (chained) timestamp: uint64 // monotonic, hardware-attested actor: Identity action: PolicyDecision resource: Resource context: ExecutionContext outcome: { allowed | denied | escalated } }

The Audit Log is the sovereign system's memory. It is append-only, cryptographically chained, and publicly verifiable by any citizen. Every access, every denial, every policy change, every key rotation is recorded. An audit log that can be modified by any actor — including system administrators — is not an audit log. It is a narrative controlled by whoever holds write access.

Primitive 06 — DataFlowPolicy
DataFlowPolicy { source_domain: string // originating identity or data domain target_domain: string // destination domain permitted_fields: string[] // fields authorized to cross the boundary correlation_mode: enum // prohibited | explicit | quorum_required authorized_by: Identity[] // who approved this cross-domain flow expiry: uint64 // time-bounded; no perpetual cross-domain grants }

Composability is the primary amplifier of control. Individually safe components — identity, behavioral data, inference outputs — become a total visibility system when joined without constraint. The DataFlowPolicy primitive is the system's answer to this: every cross-domain data join requires an explicit, authorized, time-bounded policy that names what fields may cross which boundaries and under what conditions. Unlisted joins are structurally prohibited at the pipeline boundary. No policy, no join. The sovereign system does not assume composability is safe — it requires composability to be proven safe at each instance.

The Four Enforcement Points

Sovereignty does not live in the primitives. It lives in the points where the primitives are evaluated and decisions are made. There are exactly four boundaries where enforcement is meaningful:

Ingestion Boundary

The point where external data enters the sovereign system. At ingestion, every incoming artifact — a data file, a model weight, a software package, a message — is evaluated against classification policy, provenance requirements, and integrity checks. Data whose provenance cannot be verified, software whose supply chain is unknown, and content that violates classification boundaries is rejected here. The ingestion boundary is the frontier of sovereignty: anything that passes without inspection is inside the perimeter but outside control.

Query Boundary

The point where an Identity requests access to a Resource. Identity is verified, clearance is checked against classification, and the Policy evaluator returns a decision before any data moves. Critically, the query boundary must not merely check whether the action is allowed — it must also check whether the combination of allowed queries reveals information that no single query should reveal. Aggregate access is an attack surface; the query boundary must enforce statistical disclosure controls as a policy primitive, not an afterthought.

Pipeline Boundary

The point where a computation is initiated. Every pipeline — a data transformation, a model training run, an analysis job — runs inside an Execution Context. Before execution begins, the pipeline boundary verifies that the initiating Identity has execution rights on the specified Resources, that the hardware environment is attested, and that the lineage of input Resources is consistent with the requested output classification. A pipeline that consumes classified inputs cannot produce public outputs without an explicit, audited declassification decision by an authorized Identity.

Output Boundary

The point where results leave the sovereign system. Outputs are classified by inheriting the classification of their highest-classified input, unless explicitly declassified. Every output leaving the system boundary is logged with the full Execution Context that produced it. Outputs destined for external parties carry the sovereign node's cryptographic signature, establishing provenance and enabling downstream systems to verify that the output has not been tampered with after leaving the boundary.

The Convergence Constraint

The most dangerous failure mode in a sovereign system is not external attack. It is the quiet convergence of three roles under a single authority: identity issuance, policy authorship, and enforcement. When one actor controls all three, the system is totalitarian in the structural sense — not as a policy outcome but as an architectural one. The enforcement points described above are only meaningful if no single actor can simultaneously control who has an identity, what the policies say, and whether policies are applied.

The convergence constraint mandates cryptographic separation of these roles. The separation is not procedural — it cannot be enforced by policy alone, because whoever controls the policy layer can override any policy, including policies about role separation. The separation must be enforced cryptographically: distinct key sets, held by different people selected through different mechanisms, with no MPC ceremony defined for cross-role operations.

Role 01 — Identity Authority
IdentityAuthority { key_set: MPC_KeySet // distributed across independent holders can: [ issue_credentials, revoke_credentials, rotate_keys ] cannot: [ author_policy, version_policy, operate_enforcement_runtime ] quorum: threshold_t_of_n // requires t of n keyholders to act }

Issues and revokes verifiable credentials. Holds no policy-writing capability and cannot operate the enforcement runtime. Holders of Identity Authority keys are structurally prohibited — not just procedurally prohibited — from also holding Policy or Enforcement keys. The selection mechanism for key holders must be independent of the mechanisms for the other two roles.

Role 02 — Policy Authority
PolicyAuthority { key_set: MPC_KeySet // different holders than IdentityAuthority can: [ author_policy, version_policy, deprecate_policy ] cannot: [ issue_credentials, revoke_credentials, operate_enforcement_runtime ] quorum: threshold_t_of_n // quorum set cannot overlap with other role sets }

Authors, versions, and deprecates policy. Cannot determine who holds an identity and cannot operate the enforcement runtime. Policy changes require quorum of this key set. The quorum holders are selected by a mechanism independent of how Identity Authority and Enforcement key holders are selected — preventing a single political process from staffing all three roles simultaneously.

Role 03 — Enforcement Runtime
EnforcementRuntime { key_set: MPC_KeySet // different holders than both above can: [ evaluate_policy, execute_lifecycle, write_audit_entry ] cannot: [ issue_credentials, author_policy, modify_audit_entry ] quorum: threshold_t_of_n }

Evaluates policy at all four enforcement boundaries and writes to the audit log. Cannot issue identities, cannot author policy, and — critically — cannot modify the audit entries it writes. A runtime that can erase its own decisions is not an enforcement runtime; it is a cover story. The append-only audit log enforces this: the Enforcement Runtime has write-new-entry access only, never modify-existing-entry access.

Why Structural Separation Cannot Be Procedural

A policy that says "Role X cannot do Y" can be changed by whoever controls the Policy Authority. A law that says the same can be repealed by whoever controls the legislature. Procedural separation is always one captured actor away from collapse. The convergence constraint is enforced at the cryptographic level: the three key sets do not share holders, and the system defines no operation that requires all three key sets to sign together. There is no "emergency override" ceremony. There is no "root access" credential that bypasses role separation. Any system that includes such a mechanism has not implemented the convergence constraint — it has documented a plan to eventually violate it.

Model Constraints

Inference systems — models that observe identity, behavior, and resource access to produce decisions or recommendations — occupy a special position in the sovereignty architecture. An uncontrolled model is a fourth convergence point: it can synthesize information across the three separated roles without any of the role-separation constraints applying to its outputs. The governance model is only as strong as its weakest inference boundary.

Every inference system operating on system data must satisfy three constraints:

Scope declaration. At instantiation, every model declares its input scope — which resource types, which identity fields, which behavioral signals it is permitted to consume. This declaration is a policy artifact, versioned and audited. A model cannot observe inputs outside its declared scope; the pipeline boundary enforces this before execution begins.

Lineage inheritance. Model outputs inherit the classification of their highest-classified input. An inference result derived from identity data and behavioral logs cannot be treated as less sensitive than either of those inputs. Declassification requires explicit policy authorization and produces an audit entry.

Cross-domain prohibition. A model cannot simultaneously access identity primitives and behavioral data across domains without an explicit DataFlowPolicy authorizing that combination. The prohibition on simultaneous access prevents models from becoming the implicit join layer that the convergence constraint is designed to prevent. Inference that combines who someone is with everything they have done is a total visibility instrument, regardless of whether any single access was permitted.

Every interaction with the sovereign system follows a single lifecycle. The lifecycle is not a recommendation — it is the system boundary. Any interaction that bypasses any step is outside the system and therefore outside sovereignty's enforcement reach.

System Layers vs. Philosophical Layers

The framework's seven philosophical layers (physical through telos) describe what sovereignty means. The system requires these to be mapped onto executable layers — ones that have APIs, enforcement points, and observable behavior.

Infrastructure Layer
Compute, storage, networking — physically located within the sovereign boundary, attested hardware, no foreign dependencies in the critical path.
→ Physical Sovereignty
Identity Layer
Pairwise domain-scoped identifiers, hardware-bound keys, verifiable credentials, lifecycle management, revocation registry. No global identifiers issued.
→ Institutional Architecture
Policy Layer
Policy-as-code runtime (OPA-class), MPC-enforced for sovereign-classified operations, version-controlled, publicly auditable policy corpus. Policy Authority key set is cryptographically distinct from Identity Authority and Enforcement Runtime key sets.
→ Institutional + Cognitive
Data Layer
Classified resource registry, lineage graph, knowledge commons protocol, data memory archive, statistical disclosure controls.
→ Knowledge Commons
Execution Layer
Attested compute environments, pipeline boundary enforcement, provenance-carrying execution contexts, output classification engine.
→ All layers in runtime

Delta vs. Existing Stacks

The sovereign system is not built in a vacuum. Existing systems solve adjacent problems. Understanding the delta clarifies what the sovereign system must build versus what it can adopt.

System What it solves Why it's insufficient The gap
Palantir Foundry Data governance, access control, lineage at enterprise scale Proprietary platform; the vendor holds root access; CLOUD Act exposure; you govern your data inside someone else's system Platform sovereignty — the enforcement layer must itself be sovereign
Snowflake / cloud DWH Scalable data storage and query with RBAC Foreign cloud infrastructure; data at rest in foreign jurisdiction; vendor dependency in critical path Infrastructure sovereignty — compute must run on attested hardware within the boundary
Google Zanzibar Globally consistent, fine-grained authorization at scale Authorization without execution sovereignty; controls who can read data but not where data is computed or what hardware runs it Execution sovereignty — policy enforcement must extend through the pipeline, not stop at the query
Self-Sovereign Identity (SSI) User-controlled identity, verifiable credentials, DID standards Identity layer without a policy runtime or enforcement infrastructure; no answer to "what does this identity allow you to do, and who verifies that?" Enforcement runtime — identity is necessary but insufficient without a policy layer that executes against it
Open Policy Agent (OPA) Policy-as-code, decoupled from enforcement point, reusable across systems Policy evaluator without a sovereign execution environment; policies run on whatever infrastructure the operator chose; operator can change the policies Governance of governance — policy changes must themselves require MPC authorization from distributed key holders

The sovereign system does not replace these tools. It provides the context in which they can operate without introducing foreign dependency into the critical enforcement path. OPA can be the policy runtime. Zanzibar-style authorization can be the query boundary engine. SSI standards can define the identity layer. The sovereign contribution is the architecture that makes these tools' outputs enforceable within a boundary that no external actor can override.

What Cannot Be Solved at the System Layer

The system model enforces policies that humans write. It does not write the policies. A captured policy layer — one controlled by an actor who has seized institutional authority — will enforce bad policy perfectly. The system model's defense against policy capture is the same as the framework's defense against institutional capture: distributed key management, multi-party computation, and a cognitively sovereign population capable of detecting when the policies no longer serve them. The system is the tool. The institutional and cognitive layers are what the tool serves.

What Makes a System Sovereign

A system is sovereign over a domain when: (1) it can make and enforce decisions within that domain without requiring permission from any external actor; (2) those decisions are made according to policies set by the population the system serves, not by the system's builders or operators; (3) the audit log that records every decision is accessible to every citizen and cannot be modified by any actor, including the system's administrators; and (4) no single actor can simultaneously control identity issuance, policy authorship, and enforcement — the three roles are cryptographically separated and independently governed.

Condition 1 requires infrastructure sovereignty: the enforcement points run on hardware the sovereign node controls, portable across providers without data exposure. Condition 2 requires institutional sovereignty: policy changes require MPC authorization from distributed key holders who do not overlap with identity or enforcement key holders. Condition 3 requires audit sovereignty: the log is public, append-only, and cryptographically chained, writable only by the Enforcement Runtime and unmodifiable by anyone. Condition 4 requires convergence resistance: the Identity Authority, Policy Authority, and Enforcement Runtime are held by distinct key sets selected through independent mechanisms.

These four conditions are the minimum viable specification for a sovereign system. A system that meets condition 1 but not condition 2 is an autocrat's fortress. A system that meets conditions 1 and 2 but not condition 3 is a governance system without accountability. A system that meets conditions 1 through 3 but not condition 4 is a system waiting for the moment when one actor becomes powerful enough to unify all three roles — at which point conditions 1 through 3 become instruments of total control rather than guarantees of sovereignty. All four conditions must hold simultaneously.
